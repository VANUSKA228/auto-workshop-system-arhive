4.3 Логика заглушки оплаты
ВАЖНО: никакого реального платёжного шлюза нет. Логика следующая:
Клиент заполняет форму с полями карты (номер, срок, CVV) — это поля только для UI, данные не сохраняются в БД;
Frontend отправляет POST /payments/stub с order_id и amount;
Backend создаёт запись в таблице payments со статусом stub_ok, НЕ обрабатывает данные карты;
Backend возвращает { success: true };
Frontend перенаправляет на страницу заявок с сообщением «Заявка успешно зарегистрирована».

6.1 Страница входа (/login)
Логика frontend:
Форма: React Hook Form + Yup-валидация (email формат, пароль мин. 6 символов);
POST /auth/login → получаем { token, user } → сохраняем в store и localStorage → redirect на /orders;
При ошибке 401 — показываем «Неверный email или пароль».

6.2 Навигационная панель (Navbar)
Логика: Navbar читает роль из authStore и условно скрывает/показывает пункты меню через проверку role === 'admin'.

6.3 Страница «Заявки» (/orders)
Логика frontend:
При монтировании компонент вызывает GET /orders/ (Master/Admin) или GET /orders/my (Client);
Фильтры применяются через query-параметры: ?status=new&status=in_progress&search=Иванов&date_from=2024-01-01;
Сортировка по колонкам: клик на заголовок меняет sort_by и sort_order в URL;
Строка таблицы при клике открывает модальное окно с деталями заявки;
Цвет Badge статуса: new → серый, in_progress → синий, in_repair → оранжевый, done → зелёный.

6.4 Форма создания заявки (/orders/new)
Логика frontend:
Валидация: React Hook Form + Yup. Поле «год» — только число, 4 цифры;
Услуги подгружаются при монтировании из GET /services/;
После валидации данные сохраняются в sessionStorage (или store) и выполняется переход на /payment;
На странице /payment данные формы отображаются в «шапке» (авто, услуга, итого).

6.5 Страница оплаты — ЗАГЛУШКА (/payment)
⚠️ ВНИМАНИЕ: Эта страница является декоративной заглушкой. Никакого реального платежа не происходит. После нажатия «Оплатить» система просто создаёт заявку.
Логика frontend:
Компонент читает данные заявки из sessionStorage/store;
Поля карты: маска через react-input-mask или аналог. Данные НИГДЕ не сохраняются и не отправляются;
При нажатии «Оплатить»:
1. POST /orders/ — создаётся заявка, возвращается order_id;
2. POST /payments/stub { order_id, amount } — создаётся запись-заглушка;
3. При успехе — redirect на /orders с toast-уведомлением «Заявка #N успешно создана»;
4. При ошибке — показывается alert «Что-то пошло не так, попробуйте ещё раз».
Кнопка блокируется (disabled + спиннер) во время выполнения запросов.

6.6 Форма обработки заявки (модальное окно — только Мастер)
Логика frontend:
Модальное окно открывается при клике «Обработать» в таблице заявок;
При сохранении: PATCH /orders/{id} с изменёнными полями;
Если выбран мастер → статус автоматически становится in_repair;
При нажатии «Готово»: PATCH /orders/{id} { status: 'done' } → после успеха модал закрывается, в строке таблицы Badge меняется на «Готово», появляется popover с контактами клиента.

6.7 Страница пользователей (/users) — только Администратор
Логика: POST /users/ для создания, PATCH /users/{id} для редактирования, DELETE /users/{id} для деактивации (is_active = false, не физическое удаление).

6.8 Страница услуг (/services)
Клиент видит страницу только для чтения. Мастер не имеет доступа.

8. Нефункциональные требования
8.1 Производительность
95% запросов к API — не более 300 мс при нагрузке 200 одновременных пользователей;
Загрузка страницы (LCP) — не более 2 секунд;
Таблица до 10 000 строк — серверная пагинация (limit/offset), по 50 строк на страницу.
8.2 Безопасность
Пароли хранятся в БД только в виде bcrypt-хэша (cost factor 12);
JWT: срок жизни access_token — 8 часов; при 401 — автоматический выход;
CORS: разрешён только домен фронтенда (в продакшне);
Все входные данные валидируются через Pydantic (backend) и Yup (frontend);
Автовыход при 30 минутах бездействия (реализуется через таймер на фронте).
8.3 Совместимость
Поддерживаемые браузеры: Chrome 110+, Firefox 110+, Edge 110+;
Адаптивный дизайн: от 360px (мобильный) до 1920px (десктоп).
8.4 Надёжность
Доступность системы — 99,5% рабочего времени;
Резервное копирование БД — ежедневно, хранение 30 дней;
Восстановление после сбоя — не более 4 часов.

  -- Роли
  CREATE TABLE roles (
      id         SERIAL PRIMARY KEY,
      name       VARCHAR(50) UNIQUE NOT NULL  -- 'client' | 'master' | 'admin'
  );

  -- Пользователи
  CREATE TABLE users (
      id           SERIAL PRIMARY KEY,
      first_name   VARCHAR(100) NOT NULL,
      last_name    VARCHAR(100) NOT NULL,
      middle_name  VARCHAR(100),
      email        VARCHAR(255) UNIQUE NOT NULL,
      phone        VARCHAR(20),
      password_hash VARCHAR(255) NOT NULL,
      role_id      INT REFERENCES roles(id) NOT NULL,
      is_active    BOOLEAN DEFAULT TRUE,
      created_at   TIMESTAMP DEFAULT NOW()
  );

  -- Классификатор услуг
  CREATE TABLE services (
      id    SERIAL PRIMARY KEY,
      name  VARCHAR(255) NOT NULL,
      price NUMERIC(10,2)
  );

  -- Заявки
  CREATE TABLE orders (
      id              SERIAL PRIMARY KEY,
      client_id       INT REFERENCES users(id) NOT NULL,
      master_id       INT REFERENCES users(id),  -- назначенный мастер
      car_brand       VARCHAR(100) NOT NULL,
      car_model       VARCHAR(100) NOT NULL,
      car_year        SMALLINT NOT NULL,
      description     TEXT,
      status          VARCHAR(30) NOT NULL DEFAULT 'new',
                   -- 'new' | 'in_progress' | 'in_repair' | 'done'
      created_at      TIMESTAMP DEFAULT NOW(),
      updated_at      TIMESTAMP DEFAULT NOW()
  );

  -- Связь заявка–услуга (многие ко многим)
  CREATE TABLE order_services (
      order_id   INT REFERENCES orders(id) ON DELETE CASCADE,
      service_id INT REFERENCES services(id),
      PRIMARY KEY (order_id, service_id)
  );

  -- Платежи (заглушка)
  CREATE TABLE payments (
      id              SERIAL PRIMARY KEY,
      order_id        INT REFERENCES orders(id) UNIQUE,
      card_last4      VARCHAR(4),   -- последние 4 цифры — только для UI
      amount          NUMERIC(10,2),
      status          VARCHAR(20) DEFAULT 'pending',  -- 'pending' | 'stub_ok'
      created_at      TIMESTAMP DEFAULT NOW()
  );
  CREATE INDEX idx_orders_client  ON orders(client_id);
  CREATE INDEX idx_orders_status  ON orders(status);
  CREATE INDEX idx_orders_created ON orders(created_at DESC);
  CREATE INDEX idx_users_email    ON users(email);

  # dependencies.py
  def get_current_user(token: str = Depends(oauth2_scheme), db: Session = Depends(get_db)):
      payload = jwt.decode(token, SECRET_KEY, algorithms=[ALGORITHM])
      user = db.query(User).get(payload['user_id'])
      if not user: raise HTTPException(401)
      return user

  def role_required(*roles):
      def checker(user = Depends(get_current_user)):
          if user.role.name not in roles:
              raise HTTPException(403, 'Forbidden')
          return user
      return checker

  // store/authStore.ts  (Zustand)
  interface AuthStore {
    token: string | null
    user: { id: number; name: string; role: 'client'|'master'|'admin' } | null
    setAuth: (token: string, user: AuthStore['user']) => void
    logout: () => void
  }
  // После логина: token сохраняется в localStorage, user — в store
  // Axios-интерцептор автоматически добавляет Bearer токен к каждому запросу
  // api/axiosInstance.ts
  const api = axios.create({ baseURL: import.meta.env.VITE_API_URL })

  api.interceptors.request.use(config => {
    const token = localStorage.getItem('token')
    if (token) config.headers.Authorization = `Bearer ${token}`
    return config
  })

  api.interceptors.response.use(
    res => res,
    err => {
      if (err.response?.status === 401) {
        localStorage.removeItem('token')
        window.location.href = '/login'
      }
      return Promise.reject(err)
    }
  )
